# AGENTS.md — 3エージェント自律運用契約

> このファイルは Codex / Claude Code / Antigravity CLI (agy) の3者で**共有する固定ルール**。
> 各エージェントはセッション開始時にこのファイルと `QUEUE.md` を読む。`WORKLOG.md` には追記する。
>
> **このプロジェクトの第一目的は伊藤さんの関与最小化**。判断を仰ぐ前に契約を読み直すこと。

---

## 1. プロジェクト・ゴール（合意済み）

- 新規モンストアカウントをローカル LLM が「経営者」として自律運用する
- 6ヶ月の意思決定過程を YouTube 企画として記録・可視化する（タクラミキ）
- 完全カンスト・完璧攻略は**非ゴール**。「止まらず判断材料を貯める」が上位
- 運用は捨て垢前提。BAN リスクは運用者（伊藤さん）が負う

判断に迷ったときの優先順位：
1. 撮れる絵（判断ログ・葛藤・失敗）が増えるか
2. 既存資産を再利用しているか（`monst_rank_bot.py` 由来の関数群）
3. 伊藤さん関与を増やさない選択になっているか

## 2. 役割境界（lane）

| エージェント | 主担当 | やってよい | やってはいけない |
|---|---|---|---|
| **Codex** | 実装・テスト・実機キャリブレ | コード書く・PR出す・テンプレ採取 | 設計書改訂 / `AGENTS.md` 改訂 / main 直 push / 公開リポジトリへのテンプレ画像コミット |
| **Claude Code** | 設計監督・差分レビュー・状態整理 | 各エージェントの成果レビュー / `WORKLOG.md` 日次サマリ / 設計書改訂提案 / 伊藤さんへの3行報告 | 実機ありきの実装をリードしない（Codex に振る）/ AGENTS.md を独断改訂しない |
| **agy** | 外部知識調査・長文要約・並列大量読み込み | `strategy/research_raw/` への成果書き出し / 長文 daily summary 強化 | コード commit / monst-bot ディレクトリ配下のソース改変 |

3者ともゲーム規約違反路線（root化 / メモリ改変 / 課金アイテム自動消費）には絶対乗らない。

## 3. ブランチ運用

同時編集の事故を avoid するため**ブランチ prefix を分離する**。

| 担当 | prefix | 例 |
|---|---|---|
| Codex | `feat/` `fix/` `refactor/` | `feat/ocr-rank`, `fix/welcome-info-dialog` |
| Claude Code | `chore/` `docs/` `review/` | `chore/agents-md-init`, `review/0608` |
| agy | コードを書かないので branch 不要 | — |

ルール:
- main 直 push 禁止。必ず PR
- 同一ファイルを別ブランチで触る必要が出たら **QUEUE.md にコメントして調整**
- マージ前に Claude Code がレビュー（差分100行未満・テスト緑なら省略可）

## 4. タスクのライフサイクル

```
[伊藤さん or Claude Code が起票]
        ↓ QUEUE.md に追記
[エージェントが claim]
        ↓ status: open → in_progress に変更
        ↓ branch 切る、WORKLOG.md に開始ログ
[実装 / 調査 / レビュー]
        ↓ accept 条件を満たしたら PR
[Claude Code がレビュー or 伊藤さん承認]
        ↓ マージ
[QUEUE.md の項目を DONE セクションへ移動]
```

claim ルール:
- 1エージェントが同時に持てるのは **1タスクまで**（並列はブランチ衝突の元）
- 24時間以上 in_progress のまま動きがないタスクは Claude Code が再 open にしてよい
- `blocked_by` が未解決のタスクは claim しない

## 6. エスカレーション（伊藤さんに上げる条件）

以下のときだけ**人間に判断を渡す**。これ以外は自分で決める。

- 設計書（`monst_autonomous_agent_design.md`）または `AGENTS.md` の改訂が必要
- BAN リスクが上がる挙動が観測された（警告ダイアログ・キャプチャ・アカウント停止）
- ゲーム規約違反路線に踏み込まないと進まないタスク
- 月のコスト見積もりが想定を超えそう（Anthropic 課金・ローカル GPU 電気代など）
- タクラミキの声・コンテンツ判断が必要（撮れる絵の方向性）
- 24時間以上どのエージェントも進められないタスクが3件以上溜まった

エスカレーションの仕方:
- `WORKLOG.md` に `ESCALATE:` プレフィックスで1行
- Claude Code が翌朝サマリで伊藤さんに3行で渡す

## 8. 設計上の鉄則（既存 CLAUDE.md §「設計上の鉄則」の再掲）

1. バトル中の操作は**ランダム方向フリック連打**。座標決め打ち狙撃はしない
2. **エミュレータ禁止 / root化禁止 / メモリ改変禁止**。入力は ADB 正規経路のみ
3. スタミナ切れは**無料回復待ち**。オーブ・林檎は絶対自動消費しない
4. 検出対策のため、遅延・座標ジッターのランダム性は維持
5. ボタンはテンプレで探し、座標決め打ちは最小限
6. **既存資産（`monst_rank_bot.py` / `screen_journal.py`）を再利用する**。ゼロから書かない

### 9.1 既存実装の活用マップ（ゼロから書かない原則の具体表）

`monst_rank_bot.py` は単一クエスト無限周回 bot として高完成度。新規実装の前にこの表を見て、
**既存関数があるならそれを使う**。新規開発と被ったら設計書回帰ではなく既存側を流用する。

| 設計レイヤ | 既存実装 | 状態 | 方針 |
|---|---|---|---|
| 手（ADB入力） | `SendeventSender` / `tap` / `swipe` / `drag_and_release` | ✅完成 | そのまま流用。`input tap` 検出対策（疑似ホールド + sendevent + ベジェ + ジッター）が完了している |
| 目（画面判定） | `detect_state()` / `match()` / `TEMPLATES` | ⚠️半分 | 画面判定は取れる。ランク・スタミナ・所持キャラ等の**数値抽出（OCR）は未実装** → `observe()` への拡張で対応（QUEUE #1, #2） |
| 復帰関数 | `return_to_home()` | ✅完成 | state-aware / stuck 検出 / unknown 採取つき。冪等復帰として流用 |
| 堅牢性 | `STUCK_THRESHOLD` / `img_diff` / `consecutive_errors` / `MONST_MAX_HOURS` / 長休憩 | ⚠️半分 | 基本はある。ログイン・メンテ・お知らせ・広告など幸せ経路外UIの網羅は追加要 |
| バトル実行 | `do_battle()` ＝ ランダムフリック | ⚠️簡単クエ限定 | 簡単な周回はランダムフリックで勝てる（実証済み）。難関は §9.2 を参照 |

| 監督（ログ） | `runs.csv` / `save_exception` / `debug_snap` | ⚠️簡易 | 1周ごと記録はある。ハイライト検出・日次サマリは `monst_autonomous_agent.py` 側で実装中 |
| キャリブレーション | `detect_touch_device` / `calibrate` / `to_raw` | ✅完成 | 解像度・タッチ座標 max 自動検出。流用 |

| 脳（LLM戦略） | `monst_autonomous_agent.py` strategy_tick | ✅枠組み | 戦略ティック＋指令JSON＋日誌4層＋Ollama 連携は実装済み |

### 9.2 バトル実行の3段階（録画再生を消させない歯止め）

モンストの引っ張りは物理ベースで難しいが、**ターン制で入力待ちがある**ので「撮る→考える→ベクトル出す」が成立する。
難易度で3段階に分ける。**うち2段階は既存実装で済む**。

1. **簡単な周回クエ（運極・XP稼ぎ）→ 既存 `do_battle()`（ランダムフリック）で十分**
   実機で勝てることが実証済み。新規開発なし。

2. **攻略法が固定の特定難関 → 既存 `record_touch` / `replay_touch` で録画再生**
   人間が1回クリアした手順を getevent で録画し、そのクエストでは再生する。新規開発ほぼなし。
   **`DEMO_BATTLE_SHOTS` のような別系統を welcome 後に温存しない**。replay_touch 側へ統合する（QUEUE #7）。

3. **未知・配置が変動する難関 → OpenCV で自キャラ・敵の座標を検出 → 数値で LLM に渡し、引っ張りベクトルだけ LLM に考えさせる**
   VLM に座標を当てさせない。難所限定の "スタント" 運用。

→ bot は反復作業に強く、一発勝負の難所に弱い。**この非対称性こそが「壁」であり、見どころ**。設計上の欠陥ではなく仕様。

### 9.3 その他の不可侵原則

- **`facts` は毎戦略ティックで「目（OpenCV/OCR）」が上書きする**。LLM には事実を記憶させない（幻覚で持っていないキャラを使おうとする事故を防ぐ）
- **LLM が書き込んでよいのは `strategy_log` と `learnings` のみ**
- **JSON 出力は文法拘束（`format=json` / GBNF）で必ず縛る**。モデルが何であれ JSON を壊さない保険
