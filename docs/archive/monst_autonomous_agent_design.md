# 自律モンストエージェント 設計書（実装ハンドオフ用）

> このドキュメントはコーディングエージェント（Codex / Claude Code）への引き渡しを前提とした仕様書です。
> 目的：**新規モンストアカウントを1台のローカルLLMが「経営者」として運用し、複数の長期目標を自律的に追わせる**。YouTube企画として、その意思決定の過程を記録・可視化する。

---

## 0. 前提・非ゴール

- **使い捨て前提の新規アカウント**を使う。自動操作はモンストの規約に反するためBANリスクがあるが、本企画は実験・コンテンツ目的であり、アカウントは失われても良いものとして扱う。
- **課金なし**。無料石・無料スタミナ・配布リソースのみで運用する。
- **稼働は1日8時間**。スタミナ回復が自然な行動量ゲートになる。
- 実機（Android / Pixel 8a 等）を **ADB** 経由で操作する。エミュレータではなく実機を想定。
- **非ゴール**：完璧なバトルAI／全クエスト攻略／人間並みの引っ張り精度。難所での失敗は許容し、むしろ記録対象とする。

### 用語（実装者向けの最小グロッサリ）
- **ランク**：XPで上がるアカウントレベル。上がるとスタミナ上限が増える。
- **スタミナ**：行動資源。時間で回復し、上限はランク依存。
- **運極（うんごく）**：あるモンスターのラック値を最大化した状態。同キャラを多数集めて合成して作る。＝特定の降臨クエストを反復周回して入手数を稼ぐ作業。
- **ノマクエ（ノーマルクエスト）**：メインのクエスト群。各1回クリアで進行、難易度は段階的に上がる。
- **降臨クエスト**：特定モンスターがドロップするイベント系クエスト。運極ターゲットの供給源。
- **引っ張り**：バトルの基本操作。キャラをドラッグして離すと物理で跳ね回る。**ターン制**（入力待ちが発生する）。

## ★ 既存実装（monst_rank_bot.py）の活用マップ ★ 最重要

**ゼロから作らない。** 既存の `monst_rank_bot.py` は単一クエスト無限周回botとして高完成度で、**この設計書のv0はほぼ完成済み**、v2の一部も実装済み。これを土台に「脳」と「ナビ層」を載せる。

| 設計レイヤ | 既存実装 | 状態 | 方針 |
|---|---|---|---|
| 手（ADB入力） | `SendeventSender` / `tap` / `swipe` / `drag_and_release` | ✅完成 | **そのまま流用**。`input tap`が検出で弾かれる問題を「`input swipe`疑似ホールド＋sendevent生イベント＋ベジェ曲線＋ジッター」で解決済み。最難関の手の層が終わっている |
| 目（画面判定） | `detect_state()` / `match()` / `TEMPLATES` | ⚠️半分 | 「今どの画面か」は取れる。**ランク値・スタミナ値・所持キャラ・制覇リスト等の"数値抽出(OCR)"は未実装** → §5 facts のために `observe()` へ拡張が必要 |
| 復帰関数 | `return_to_home()` | ✅完成 | §7の冪等復帰そのもの。state-aware・stuck検出・unknown画面の自動採取つき。流用 |
| 堅牢性 | `STUCK_THRESHOLD` / `img_diff` / `consecutive_errors`自動停止 / `MONST_MAX_HOURS` / 人っぽい長休憩 | ⚠️半分 | 基本はある。ログイン・メンテ・お知らせ・広告など幸せ経路外UIの網羅は追加要 |
| バトル実行 | `do_battle()` ＝ ランダムフリック | ⚠️簡単クエ限定 | 簡単な周回はランダムフリックで勝てる（実証済み）。難関ノマは勝てない＝§6の課題そのもの |
| マクロ記録/再生 | `record_touch` / `replay_touch`（getevent） | ✅完成 | 特定難関を人間が1回クリア→録画→再生できる。§6の「録画マクロ」路線が既にある |
| 監督（ログ） | `runs.csv` / `save_exception` / `debug_snap` | ⚠️簡易 | 1周ごと記録はある。§8のハイライト検出・ダッシュボード・日誌は未 |
| キャリブレーション | `detect_touch_device` / `calibrate` / `to_raw` | ✅完成 | 解像度・タッチ座標maxの自動検出つき。流用 |
| 脳（LLM戦略） | — | ❌未 | 新規。戦略ティック＋指令JSON＋日誌 |
| クエスト選択/ナビ | `super_shortcut` 固定 | ❌未 | **最大の改修点**。今は1クエスト固定。任意クエスト（運極の降臨／各ノマ）へ到達するナビ層が要る |
| デッキ管理 | `deck.png`/`target_deck.png` テンプレのみ | ❌ほぼ未 | `manage_team` のため編成選択・変更が要る |

### 中心となる改修は1点
現在の `run_one_cycle()` は `super_shortcut → 出撃 → バトル → ホーム` の**固定フロー**。LLM版では、これを **(a) 指令（クエスト・デッキ・停止条件）でパラメータ化された実行関数** と **(b) 任意クエストへ到達するナビゲーション層** に作り替える。ここが作業の主軸。

### うれしい誤算：統合点は素直
`run_one_cycle()` の戻り値 `(result, note)`（`ok` / `stamina_out` / `stuck` / `timeout`）が、**§3のエスカレーション・トリガーとほぼ1:1で対応している**。今は `run()` がこの結果を見て `time.sleep(STAMINA_WAIT)` 等を**決め打ちで反応**している。この決め打ち反応部分を**LLM戦略ティック呼び出しに差し替えるだけ**で「実行ティック→エスカレーション→戦略ティック」構造が成立する。見た目より統合は簡単。

## 1. 目標（複数同時に追う）

| 目標 | 種類 | 性質 |
|------|------|------|
| ランク100到達 | XP蓄積 | どのクエでもいい。スタミナあたりXP効率が指標 |
| 運極作成 | 反復周回 | 1つの降臨クエを反復。bot最得意。ターゲット選定はLLM判断 |
| ノマクエ制覇 | 段階攻略 | 各1回クリア。難易度上昇＝**チーム強化が前提条件** |

### 相互依存（このエージェントの戦略の核）
3目標は**同じスタミナを奪い合う**上に依存関係がある：
- ノマクエはいずれ「今の編成では勝てない壁」にぶつかる。
- 壁突破には強化が必要 → 運極キャラ育成 or 配布リソースでの強化に依存。
- 運極周回・ノマ攻略はどちらもXPを生み、ランク進行に寄与する。

→ LLMには「壁→運極周回→突破」のような**優先順位の組み替え**を判断させる。これが企画の見どころになる。

## 2. 全体アーキテクチャ（目 / 脳 / 手 / 記憶 / 監督）

```
┌──────────────────────────────────────────────┐
│  監督 (Director): ログ / ハイライト検出 / ダッシュボード │
└──────────────────────────────────────────────┘
        ▲ 全イベントを記録
        │
[目 OpenCV/OCR] → [脳 ローカルLLM] → [手 ADB] → 実機
   状態抽出          方針(指令)出力     タップ/スワイプ
        │                ▲
        └─────[記憶 日誌DB]─────┘
              脳が読み書きする
```

- **目（OpenCV / OCR）**：画面から構造化データを抽出する。決定論的で安定。VLMは原則使わない（既存スキルで賄え、判定がブレない）。
- **脳（ローカルLLM）**：目が出した構造化状態＋目標＋日誌を受け取り、**指令(JSON)** と理由（日本語）を返す。
- **手（ADB）**：指令をタップ・スワイプに変換して実行。
- **記憶（日誌DB）**：脳が読み書きする永続メモリ（§5）。
- **監督（Director）**：全判断と結果を記録し、コンテンツ素材を生成（§7）。

## 3. 二重ループ構造

LLMを**毎バトル呼ばない**。LLMは方針を決める係、スクリプトは方針を実行する係に分離する。

```
[戦略ティック / 遅い / LLM]
  日誌 + 現状スナップショット + 目標台帳 を読む
  → 指令 directive を1つ出す（例: クエストXを20回周回）
        ↓
[実行ティック / 速い / スクリプト]
  指令クエストを OpenCV + ADB でループ実行
  （入場 → バトル → 報酬回収 → 繰り返し）
  → 終了条件 or 例外 でLLMに戻す（エスカレーション）
```

### エスカレーション・トリガー（LLMを呼び戻す条件）
- 指令完了（指定回数を消化した）
- スタミナ切れ（待つ / 石を使う / 目標を切り替える を判断）
- 新キャラ入手（保持？運極の餌？編成変更？）
- クエスト失敗（リトライ / 強化が先 / 諦める を判断）
- マイルストーン到達（運極完成・ランクアップ・ノマクエ制覇）
- スタック復帰後（§6）

## 4. LLM指令の出力スキーマ

戦略ティックでLLMに渡す入力：
`3目標と各進捗（台帳）` + `事実スナップショット（目が生成）` + `直近の日誌数件（全件は入れない）` + `今呼ばれた理由（トリガー）` + `選択可能クエスト一覧（スタミナ消費・ドロップ付き）`

出力は**JSON固定**。スクリプトは `action` で分岐する：

```json
{
  "diary_entry": "ノマクエ◯◯で2連敗。火力不足。先に△△の運極を進めて編成を補強する",
  "reasoning": "壁を無理に殴るよりスタミナ効率の良い周回に振った方が長期的に得",
  "directive": {
    "action": "farm_quest",
    "target": "△△降臨【究極】",
    "team": ["A", "B", "C", "D"],
    "stop_condition": { "type": "repeat", "count": 20 },
    "priority_goal": "ungoku"
  }
}
```

- `action`：`farm_quest` / `attempt_clear` / `feed_luck` / `manage_team` / `wait_stamina` / `pull_gacha`
- `stop_condition.type`：`repeat`（count） / `stamina_below`（value） / `item_collected`（item, count）
- `priority_goal`：`rank100` / `ungoku` / `noma`

**JSONは文法拘束（Ollama `format=json` か GBNF）で必ず縛ること。** モデル選びより信頼性に効く。

## 5. 日誌（記憶）スキーマ

日誌は単なるログではなく、**AIが読み返して賢くなるフィードバック装置**。4層に分ける。

1. **目標台帳（goal_ledger）**：3目標の進捗と「今の障害」。例：`noma: 18/◯◯ クリア、次の◯◯で詰まり中`
2. **戦略ログ（strategy_log, 追記式）**：日付つきの「今の方針と理由」。→ ナレーション素材。
3. **学び・メモ（learnings）**：「クエAはスタミナ効率が良い」「編成Bで◯◯に勝てた/負けた」。→ **これを読み返すから"学習してるAI"に見える**。
4. **事実スナップショット（facts）**：ランク・スタミナ・主要キャラ・ノマ制覇リスト。

### 厳守ルール
- **`facts` は毎戦略ティックで「目（OpenCV）」が上書きする。** LLMには事実を記憶させない（幻覚で持っていないキャラを使おうとする事故を防ぐ）。
- **LLMが書き込んでよいのは `strategy_log` と `learnings` のみ。**
- 1日の終わりに「今日のまとめ」を1エントリ書かせる（やったこと・達成・所感・明日の方針）。→ 動画の1セグメントになる。

## 6. 引っ張り（バトル実行）の扱い

モンストの引っ張りは物理ベースで難しいが、**ターン制で入力待ちがある**ため「撮る→考える→ベクトル出す」が成立する。難易度で3段階に分ける。**うち2段階は既存実装で済む。**

1. **簡単な周回クエ（運極・XP稼ぎ）→ 既存の `do_battle()`（ランダムフリック）で十分。** 実機で勝てることが実証済み。新規開発なし。
2. **攻略法が固定の特定難関 → 既存の `record_touch` / `replay_touch` で録画再生。** 人間が1回クリアした手順をgeteventで録画し、そのクエストでは再生する。新規開発ほぼなし。
3. **未知・配置が変動する難関 → OpenCVで自キャラ・敵の座標を検出 → 座標を数値でLLMに渡し、引っ張りベクトルだけLLMに考えさせる**（VLMに座標を当てさせない）。ここだけが新規。難所限定の"スタント"運用。

→ 結果として **botは反復作業に強く・一発勝負の難所に弱い**。この非対称性こそが「壁」であり、見どころになる。設計上の欠陥ではなく仕様。`do_battle()` の「方向で挙動が変わらないからランダムでいい」という前提は簡単クエ限定であり、難関では3.に切り替えること。

## 7. 堅牢性（実装工数の8割はここ）

- **幸せ経路外UIの網羅**：広告・ログイン・メンテナンス・お知らせポップ・「まだ見てますか」系・ネットワークエラー・通信切断。テンプレートマッチで検出→閉じる処理を1つずつ潰す。（既存は固定セットのみ対応。新規アカウントの初回チュートリアル/お知らせ系を要追加。）
- **ウォッチドッグ**：同一画面がN分続いたらスタック判定 → アプリ強制再起動 → 既知の初期画面まで復帰 → エスカレーション。（既存に `img_diff` によるstuck検出と `consecutive_errors` 自動停止あり。アプリ強制再起動からの自動復帰は要追加。）
- **冪等な復帰**：どの画面からでも「ホーム」に戻れる復帰関数を1本用意し、例外時は必ずそこへ。（**既存 `return_to_home()` がこれ。流用。**）

## 10. ローカルLLM選定

**前提：戦略ティックは遅いループ（エスカレーション時のみ起動）。よって速度より信頼性と日本語品質を最大化する。**

- **常用（指令JSON生成）：Qwen3.6 9B（Q5_K_M / 32Kコンテキスト / reasoning off）。**
  - ツールコール・JSON出力の安定性が高い。16GBにフル搭載可能。日本語が強い。
  - tool callが不安定な場合は reasoning を false に固定する。
- **日誌「今日のまとめ」用（任意）：reasoning ありモデルを別建て（例：GPT-OSS 20B）。**
  - 思考トレースがそのまま「AIの葛藤」コンテンツになる。1日1回起動ならVRAM入れ替えコストは無視できる。
- **必須：出力は文法拘束（`format=json` / GBNF）で縛る。** モデルが何であれJSONを壊さないための保険。

> 注：ローカルLLMの最適解は変動が速い。確定前に実環境で日本語出力を一度確認すること。

## 11. 段階的ビルド順（既存 monst_rank_bot.py からの差分）

- **v0 ハーネス：★ほぼ完成済み（monst_rank_bot.py）。** ADB接続・画面キャプチャ・テンプレマッチ・`return_to_home`・1クエスト周回ループ・watchdog基礎・BAN対策が実装済み。残作業は新規アカウントの初期画面（チュートリアル/初回お知らせ）の確認程度。
- **v1 脳の接続：**
  - `detect_state()` を `observe()` へ拡張し、§5 `facts`（ランク値・スタミナ値・主要キャラ・制覇リスト）を**数値でOCR抽出**する。
  - 戦略ティック（LLM）＋日誌4層（SQLite/JSON）＋指令JSONスキーマを新規実装。
  - `run_one_cycle()` を**指令でパラメータ化**（クエスト・デッキ・停止条件を引数化）。
  - `run()` の決め打ち反応（`time.sleep(STAMINA_WAIT)` 等）を**LLM戦略ティック呼び出しに差し替え**。`(result,note)` をトリガーとして渡す。
- **v2 ナビ＆堅牢化：**
  - **ナビゲーション層**を新規実装（`super_shortcut` 固定をやめ、任意の降臨／各ノマクエへ到達できるように）。`manage_team`（編成選択）もここ。
  - エスカレーション・トリガー全実装（新キャラ入手・クエスト失敗の検出を追加）。
  - 幸せ経路外UIの網羅、アプリ強制再起動からの自動復帰。8時間連続稼働を達成。
- **v3 難所＆コンテンツ：**
  - 難関の `replay_touch`（録画再生）と、未知難関のLLM補助引っ張り（§6-3）。
  - Director：ハイライト検出・進捗ダッシュボード・「今日のまとめ」日誌エクスポート（既存 `runs.csv` を拡張）。

## 12. オープンクエスチョン（実装着手前に詰める）

- 運極の初回ターゲットをどう選ばせるか（候補クエストの供給と難易度をどう `facts` に載せるか）。
- スタミナ回復待ちの扱い（8時間枠内でアイドルするか、別目標に切り替えるか）。
- ノマクエの「勝てない壁」判定基準（何連敗で `attempt_clear` を諦めて強化に回るか）。
